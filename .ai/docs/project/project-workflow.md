# 프로젝트 워크플로우

> 이 문서는 이 API 프로젝트의 설계·구현·테스트 워크플로우를 정의합니다.  
> 여러 프로젝트에서 공통 개발 프로세스로 재사용할 수 있습니다.

---

## 1. 전체 흐름

1. API 설계
2. API 설계 문서 작성 (`docs/api-design`)
3. 테스트 시나리오/테스트 문서 작성 (`docs/api-tests`)
4. 설계/테스트 문서 검토 및 확정
5. 코드 스캐폴딩 및 구현 (`src/main/java`)
6. 테스트 코드 작성/자동화 (`src/test/java`)

---

## 2. 단계별 원칙

### 2.1 API 설계

- 코드 구현보다 **문서 기반 설계를 우선**한다.
- URI, JSON, 패키지 네이밍은 `docs/naming/naming-rules-<domain>.md` 에 정의된 규칙을 따른다.
- 초기 단계에서 테스트용 API를 만들더라도,  
  실제 도메인으로 확장하기 쉬운 구조와 패턴을 사용한다.

### 2.2 API 설계 문서 (`docs/api-design`)

- 개요와 리소스별 설계를 분리한다.
  - `api-design-<domain>-overview-v1.md`
  - `api-design-<domain>-resource-<resource>-v1.md`
- 파일명에 버전(v1, v2, …)을 포함해 변경 이력을 관리한다.
- 하나의 리소스는 하나의 설계 문서에 정리한다.

### 2.3 테스트 문서 (`docs/api-tests`)

- 각 엔드포인트에 대해:
  - 정상 케이스
  - 오류 케이스
  - 경계/엣지 케이스
  를 명시적으로 정의한다.
- 테스트 문서는 테스트 코드 생성 및 검토의 **단일 기준**이 된다.

### 2.4 코드 스캐폴딩 및 구현

- 설계/테스트 문서가 확정된 엔드포인트부터 구현한다.
- Spring Boot 등 프레임워크의 권장 구조를 따른다.
- Controller/Service/Domain/Infrastructure 등의 레이어를 명확히 분리한다.

### 2.5 테스트 코드 작성/자동화

- `src/test/java` 아래에 API 테스트를 포함한 단위/통합 테스트 코드를 작성한다.
- 가능한 경우 CI 파이프라인에 테스트를 통합해 자동 실행한다.

---

## 3. 테스트용 API에 대한 규칙 (선택)

- 초기 설계/구조 검증을 위해 테스트용 API를 만들 수 있다.
  - 예: `/api/health`, `/api/samples`
- 테스트용 API 역시:
  - 설계 문서 → 테스트 문서 → 코드 → 테스트 코드 순서를 따른다.
- 실제 도메인 설계가 완료되면:
  - 테스트용 API는 삭제하거나 별도의 샘플 모듈로 분리한다.
